# ADR-001: Translation Layer vs SpecKit Modification

> **Navigation**: [Index](../../readme.md) | **Prev**: [Workflows](../workflows.md) | **Next**: [Folder Structure](../../03-components/folder-structure.md)

---

## Status

**DECIDED** (2025-11-28)

## Context

Sprints generated by SpecKit produce granular, research-backed tasks (e.g., 56 tasks for a feature). However, these tasks need more explicit, detailed instructions for implementor agents to prevent implementation theater.

Two approaches were considered:
- **Option A**: Modify SpecKit to generate enhanced output directly
- **Option B**: Orchestrator translates standard SpecKit output into explicit instructions

## Decision

**Implement Option B - Orchestrator Translation Layer**

## Rationale

### Why Translation Layer (Option B)?

1. **"We don't know what we don't know"**
   - Sprint requirements often change based on implementation challenges
   - Orchestrator can adapt translations based on what's discovered
   - Rigid pre-generated specs can't incorporate runtime learnings

2. **Runtime adaptability**
   - If Task 5 fails, orchestrator can adjust Task 6's instructions
   - Failed approaches can be documented and avoided in future tasks
   - Codebase state at translation time is current, not stale

3. **Codebase context injection**
   - Orchestrator analyzes ACTUAL codebase state at translation time
   - Existing patterns and conventions discovered fresh
   - File paths verified to exist before handover

4. **SpecKit stays clean**
   - No coupling to our specific orchestrator pattern
   - SpecKit remains useful for human developers too
   - Other teams can use different translation approaches

5. **Model flexibility**
   - Can adjust translation detail level based on implementor model tier
   - More detail for lower-tier models, less for capable ones
   - Same SpecKit output, different translation depths

6. **Matches real SDLC**
   - Senior architects don't write code, they translate specs to tickets
   - Junior developers follow explicit instructions
   - Same pattern: orchestrator translates, implementor executes

### Why NOT Direct Enhancement (Option A)?

1. **Over-specification risk**
   - Details generated before codebase analysis may be wrong
   - Paths might not exist, patterns might differ
   - Requires constant spec updates as codebase evolves

2. **Coupling problem**
   - SpecKit becomes dependent on Orchestra structure
   - Changes to either require coordinated updates
   - Reduces modularity

3. **Lost flexibility**
   - Can't adapt to discovered requirements
   - Can't incorporate learnings from failed tasks
   - Rigid spec limits problem-solving

## Consequences

### Positive

- Two-step process enables runtime adaptation
- SpecKit and Orchestra remain independent
- Translation can be tailored per task complexity
- Orchestrator has full context at translation time

### Negative

- Extra processing step required
- Orchestrator quality affects translation quality
- Potential for inconsistency if orchestrator drifts
- Higher orchestrator workload

### Mitigations

- Standardized translation templates
- Pre-flight checklists for orchestrator
- Validation scripts to catch drift
- Human oversight for critical tasks

## Implementation

### Translation Process

```
SpecKit Sprint YAML          Orchestrator Translation         Implementor Task
──────────────────           ──────────────────────           ────────────────
tasks:                       For each task:                   
  - id: 1                    1. Read task description         task_001.yaml:
    title: YAxisPosition     2. Analyze current codebase        - exact file paths
    description: ...         3. Find patterns/conventions       - exact code to write
                             4. Generate explicit spec          - constraints
                             5. Add verification criteria       - no decisions needed
```

### Task Consolidation

SpecKit may produce very granular tasks. Orchestrator consolidates:

| SpecKit Tasks | Orchestrator Task | Rationale |
|---------------|-------------------|-----------|
| T001, T002, T003 | Task 1 | Same file/module |
| T004, T005 | Task 2 | Model + tests + export |
| T006 | Task 3 | Complex, keep separate |

### Translation Elements Added

For each SpecKit task, orchestrator translation adds:

1. **Codebase context**
   - Existing patterns found
   - Related files analyzed
   - Naming conventions documented

2. **Explicit file operations**
   - CREATE: exact path, suggested content/scaffold
   - UPDATE: exact path, specific changes to make
   - EXPORT: barrel file location

3. **Implementation constraints**
   - What NOT to do
   - Utilities to use (not duplicate)
   - Patterns to follow

4. **Verification hints** (for orchestrator's use)
   - Hidden from implementor
   - Adversarial checks for integration tasks
   - Visual criteria for rendering tasks
