# Spec 004.3: Athlete Datastore & Management

**Status:** Draft
**Parent:** Spec 004 (BravenLab Studio Integration)

## 1. Objective
Define the data layer that manages Athletes, Sessions, and large File Data, serving as the "memory" for the agent.

## 2. Entities (BLS Implementation)

### 2.1 Athlete
Unique identity.
- `id`: UUID
- `name`: "Hansie Joubert"
- `metrics`: { "ftp": 280, "max_hr": 195 }
- `tags`: ["cyclist", "pro"]

### 2.2 Session
A container for analysis.
- `id`: UUID
- `athleteId`: Link to Athlete
- `date`: DateTime
- `files`: List<FileReference> (Paths to .FIT/.CSV)
- `charts`: List<ChartConfig> (Saved visualizations)
- `chatHistory`: List<Message>

## 3. Data Integration Interfaces (BCP Definition)

BCP defines *what* it needs to know. BLS implements *how* it gets it.

```dart
// In BCP Package:
abstract class DataContext {
  Future<Map<String, dynamic>> getAthleteMetadata(String athleteId);
  Future<List<DataSummary>> getSessionFiles(String sessionId);
  Future<Map<String, double>> queryMetric(String fileId, String metric, TimeRange range);
}
```

## 4. FIT File Pipeline (Integration)
The pipeline described in technical debt (`fit_file_data_pipeline.md`) lives here.

1.  **Ingest:** BLS reads `.fit` file.
2.  **Process:** BLS uses a BCP utility helper to extract metadata + reduce points.
3.  **Store:**
    - **Metadata:** Saved to SQLite (fast access for Agent).
    - **Reduced Series:** Saved to SQLite (fast access for Chart Renderer).
    - **Raw Data:** Kept as `.fit` or binary blob (archive only, or on-demand deep query).

## 5. Security & Privacy
- Local-first architecture. BCP never sends raw data to the LLM.
- LLM only sees anonymized metadata or aggregate stats unless explicitly instructed otherwise.
