## 1. Objective

**Decouple intelligence from rendering.**

Create a new `braven_agent` package that:

1. Manages chat conversation state.
2. Communicates with LLMs (Anthropic initially, extensible to others).
3. Executes tools (`create_chart`, `modify_chart`) that produce `ChartConfiguration` objects.
4. Exposes a clean, UI-bindable API (`AgentSession`).

The existing `braven_chart_plus` becomes a **pure rendering library** with no AI dependencies.

## 2. Scope & Constraints

### 2.1 This is a CLEAN Implementation

**This is NOT a migration or copy of `/agentic`.**

- Build from scratch with clean architecture
- Reference `/agentic` code for patterns and working examples
- Include ONLY what is necessary for core functionality
- Do not duplicate unnecessary complexity

### 2.2 Core Requirements (MVP)

| Capability        | Description                                              |
| ----------------- | -------------------------------------------------------- |
| **Create Chart**  | LLM can create a new chart via `create_chart` tool       |
| **Modify Chart**  | LLM can update an existing chart via `modify_chart` tool |
| **Render Chart**  | `ChartConfiguration` → `BravenChartPlus` widget          |
| **Session State** | Track conversation history and active chart              |

### 2.3 Out of Scope (V1)

These exist in `/agentic` but are NOT needed for V1:

- `load_data_tool` - Data loading from files
- `describe_data_tool` - Data analysis
- `calculate_metric_tool` - Metric calculations
- `add_annotation_tool` - Annotation management
- `process_data_tool` - Data transformations
- File picker / URL fetcher services
- Data optimizer / context loader services
- Chart history / versioning
- **File attachments (FIT, CSV, etc.)** - Requires 004.3 DataContext

### 2.3.1 Attachments Scope (V1)

The `transform()` method accepts `attachments`, but V1 only supports:

| Type       | V1 Support | Notes                             |
| ---------- | ---------- | --------------------------------- |
| Images     | ✅ Yes     | Screenshots, PNG/JPEG for context |
| Text files | ⚠️ Basic   | Decoded to `TextContent` inline   |
| FIT/CSV    | ❌ No      | Requires 004.3 `DataContext`      |
| PDF        | ❌ No      | Future enhancement                |

The `AnthropicAdapter` handles supported types:

- `ImageContent` → Anthropic's vision API format
- `TextContent` → Inline text in message
- Unsupported types → Return error or ignore with warning

### 2.4 Direct Translations (Exceptions)

**ONLY these two components should be directly translated from `/agentic`:**

| Component            | Source                                         | Reason                                      |
| -------------------- | ---------------------------------------------- | ------------------------------------------- |
| `ChartRenderer`      | `lib/src/agentic/services/chart_renderer.dart` | Complex property mapping, already working   |
| Property Wiring Test | `test/agentic/property_wiring_test.dart`       | Validates all chart properties are rendered |

All other components must be implemented fresh from this specification.

### 2.5 Fallback Reference

If an implementor gets **technically stuck** on complex implementation details, they may consult `/agentic` code as a working reference. This is a **troubleshooting aid only**, NOT a design reference.

**When to consult `/agentic`:**

- Implementation is blocked on a technical detail not covered in this spec
- Need to understand how a specific API integration works

**When NOT to consult `/agentic`:**

- For design decisions (use this spec)

### 4.1.1 Serialization Methods

All domain models must support JSON serialization for persistence and API communication:

```dart
class ChartConfiguration {
  // ... fields as above ...

  /// Creates a ChartConfiguration from a JSON map.
  factory ChartConfiguration.fromJson(Map<String, dynamic> json) {
    return ChartConfiguration(
      id: json['id'] as String?,
      type: ChartType.values.byName(json['type'] ?? 'line'),
      title: json['title'] as String?,
      subtitle: json['subtitle'] as String?,
      series: (json['series'] as List)
          .map((s) => SeriesConfig.fromJson(s as Map<String, dynamic>))
          .toList(),
      xAxis: json['xAxis'] != null
          ? XAxisConfig.fromJson(json['xAxis'] as Map<String, dynamic>)
          : null,
      yAxes: (json['yAxes'] as List?)
              ?.map((y) => YAxisConfig.fromJson(y as Map<String, dynamic>))
              .toList() ??
          [],
      // ... remaining fields similarly
    );
  }

  /// Converts this configuration to a JSON map.
  Map<String, dynamic> toJson() {
    return {
      if (id != null) 'id': id,
      'type': type.name,
      if (title != null) 'title': title,
      if (subtitle != null) 'subtitle': subtitle,
      'series': series.map((s) => s.toJson()).toList(),
      if (xAxis != null) 'xAxis': xAxis!.toJson(),
      if (yAxes.isNotEmpty) 'yAxes': yAxes.map((y) => y.toJson()).toList(),
      // ... remaining fields similarly
    };
  }

  /// Creates a copy with specified fields replaced.
  ChartConfiguration copyWith({
    String? id,
    ChartType? type,
    String? title,
    // ... all fields
  }) {
    return ChartConfiguration(
      id: id ?? this.id,
      type: type ?? this.type,
      title: title ?? this.title,
      // ... all fields
    );
  }
}
```

**Implementation Rule:** All model classes (`SeriesConfig`, `XAxisConfig`, `YAxisConfig`, `AnnotationConfig`, `ChartStyleConfig`) must implement:

- `factory T.fromJson(Map<String, dynamic> json)`
- `Map<String, dynamic> toJson()`
- `T copyWith({...})`

### 5.3 Anthropic Adapter (`anthropic_adapter.dart`)

Converts between `AgentMessage` <-> `anthropic.Message`.

**Key Responsibilities:**

1. Map `AgentMessage` to Anthropic's message format.
2. Map `AgentTool` definitions to Anthropic's tool schema.
3. Parse Anthropic response back to `AgentMessage`.
4. Handle streaming (optional for V1).

#### 5.3.1 Tool Conversion Example

The adapter must convert our `AgentTool` to Anthropic's expected format:

```dart
/// Converts AgentTool list to Anthropic tool definitions.
List<anthropic.Tool> _convertTools(List<AgentTool> tools) {
  return tools.map((tool) => anthropic.Tool(
    name: tool.name,
    description: tool.description,
    inputSchema: anthropic.ToolInputSchema(
      type: 'object',
      properties: tool.inputSchema['properties'] ?? {},
      required: tool.inputSchema['required'] ?? [],
    ),
  )).toList();
}
```

#### 5.3.2 Message Conversion

```dart
/// Converts AgentMessage to Anthropic message format.
anthropic.Message _convertMessage(AgentMessage msg) {
  return anthropic.Message(
    role: msg.role == MessageRole.user
        ? anthropic.MessageRole.user
        : anthropic.MessageRole.assistant,
    content: msg.content.map(_convertContent).toList(),
  );
}

dynamic _convertContent(MessageContent content) {
  return switch (content) {
    TextContent(:final text) => anthropic.TextBlock(text: text),
    ImageContent(:final base64Data, :final mediaType) =>
      anthropic.ImageBlock(
        source: anthropic.ImageBlockSource(
          type: 'base64',
          mediaType: mediaType,
          data: base64Data,
        ),
      ),
    ToolUseContent(:final id, :final toolName, :final input) =>
      anthropic.ToolUseBlock(id: id, name: toolName, input: input),
    ToolResultContent(:final toolUseId, :final output, :final isError) =>
      anthropic.ToolResultBlock(
        toolUseId: toolUseId,
        content: output,
        isError: isError,
      ),
    BinaryContent() => throw UnsupportedError('BinaryContent not supported'),
  };
}
```

### 5.4 LLM Registry (`llm_registry.dart`)

Factory pattern for provider extensibility. See [llm-architecture.md](decisions/llm-architecture.md) Section 3.

**Rationale:** Decouples provider creation from usage. Adding a new provider (e.g., OpenAI) requires:

1. Implement `LLMProvider` interface
2. Register the factory
3. Done - no changes to `AgentSession`

````dart
/// Factory function type for creating LLM providers.
typedef LLMProviderFactory = LLMProvider Function(LLMConfig config);

/// Registry for LLM provider factories.
///
/// Usage:
/// ```dart
/// // At app startup, register providers
/// LLMRegistry.register('anthropic', (config) => AnthropicAdapter(config));
/// LLMRegistry.register('openai', (config) => OpenAIAdapter(config));
///
/// // Later, create provider by name
/// final provider = LLMRegistry.create('anthropic', myConfig);
/// ```
class LLMRegistry {
  static final Map<String, LLMProviderFactory> _factories = {};

  /// Register a provider factory.
  static void register(String id, LLMProviderFactory factory) {
    _factories[id] = factory;
  }

  /// Create a provider instance.
  /// Throws [ArgumentError] if provider ID is not registered.
  static LLMProvider create(String id, LLMConfig config) {
    final factory = _factories[id];
    if (factory == null) {
      throw ArgumentError('Unknown LLM provider: $id. '
          'Available: ${_factories.keys.join(', ')}');
    }
    return factory(config);
  }

  /// Check if a provider is registered.
  static bool isRegistered(String id) => _factories.containsKey(id);

  /// List all registered provider IDs.
  static List<String> get registeredProviders => _factories.keys.toList();
}
````

**Future providers (not in V1):**

- Google Gemini (`'gemini'`)
- Local models via Ollama (`'ollama'`)

### 5.5 Context Window Management

See [llm-architecture.md](decisions/llm-architecture.md) Section 6.

**Key Principle:** Context management lives in the **Agent Layer** (`AgentSession`), NOT the **Provider Layer** (`LLMProvider`). The provider just sends what it is given.

**V1 Approach (Simple):**

- No automatic token counting
- If history gets too long, LLM will return an error
- App handles by clearing history or starting new session

@override
Future<ToolResult> execute(Map<String, dynamic> input) async {
  try {
    // 1. Validate prompt
    final prompt = input['prompt']?.toString().trim() ?? '';
    if (prompt.isEmpty) {
      return ToolResult(
        output: 'Error: prompt is required',
        isError: true,
      );
    }

    // 2. Resolve chart type
    final chartType = _resolveChartType(input['type'], prompt);

    // 3. Build series with default colors
    final series = _buildSeries(input['series']);

    // 4. Build axes
    final xAxis = _buildXAxis(input['xAxis']);
    final yAxes = _buildYAxes(input, series);

    // 5. Build annotations
    final annotations = _buildAnnotations(input['annotations']);

    // 6. Create configuration
    final config = ChartConfiguration(
      id: Uuid().v4(),
      type: chartType,
      title: input['title'],
      subtitle: input['subtitle'],
      series: series,
      xAxis: xAxis,
      yAxes: yAxes,
      annotations: annotations,
      style: _buildStyle(input['style']),
      showGrid: input['showGrid'],
      showLegend: input['showLegend'],
      legendPosition: input['legendPosition'],
      useDarkTheme: input['useDarkTheme'],
      showScrollbar: input['showScrollbar'],
      normalizationMode: _parseNormalizationMode(input['normalizationMode']),
      width: input['width']?.toDouble(),
      height: input['height']?.toDouble(),
    );

    // 7. Return success with structured data
    return ToolResult(
      output: 'Chart created successfully with ${series.length} series.',
      isError: false,
      data: config,  // AgentSession captures this
    );

  } catch (e) {
    return ToolResult(
      output: 'Error creating chart: ${e.toString()}',
      isError: true,
    );
  }
}

      'subtitle': {'type': 'string'},

    'required': []  // No required fields - only specify what changes
  };

    if (activeChart == null) {
      return ToolResult(

        isError: true,

        isError: false,

        output: 'Error modifying chart: ${e.toString()}',
        isError: true,
      );
    }
  }

  ChartConfiguration _applyModifications(
    ChartConfiguration existing,
    Map<String, dynamic> modifications,
  ) {
    // Merge modifications onto existing config
    // Each field: use modification if present, else keep existing
    return ChartConfiguration(
      id: existing.id,  // Preserve ID
      type: modifications['type'] != null
          ? _parseChartType(modifications['type'])
          : existing.type,
      title: modifications['title'] ?? existing.title,
      subtitle: modifications['subtitle'] ?? existing.subtitle,
      series: modifications['series'] != null
          ? _mergeSeries(existing.series, modifications['series'])
          : existing.series,
      xAxis: modifications['xAxis'] != null
          ? _mergeXAxis(existing.xAxis, modifications['xAxis'])
          : existing.xAxis,
      yAxes: modifications['yAxes'] != null
          ? _mergeYAxes(existing.yAxes, modifications['yAxes'])
          : existing.yAxes,
      // ... merge all other fields similarly
    );
  }
}

### 9.4 AgentSession Interface & Implementation

See [chat-architecture.md](decisions/chat-architecture.md) for design rationale.

```dart
abstract class AgentSession {
  ValueListenable<SessionState> get state;
  Stream<AgentEvent> get events;

  /// Send a user message and get agent response.
  Future<void> transform(String prompt, {List<BinaryContent>? attachments});

  /// Update the active chart (user edited it in UI).
  /// This updates state.activeChart and emits ChartUpdatedEvent.
  /// The NEXT transform() call will include this chart as context.
  /// See [chart-propagation.md](decisions/chart-propagation.md) for flow.
  void updateChart(ChartConfiguration newConfig);

  /// Cancel any in-progress operation.
  Future<void> cancel();

  /// Clean up resources.
  void dispose();
}

class AgentSessionImpl implements AgentSession {
  final LLMProvider _llmProvider;
  final List<AgentTool> _tools;
  final String _systemPrompt;  // V1: Simple string, no context injection

  final _state = ValueNotifier<SessionState>(const SessionState());
  final _events = StreamController<AgentEvent>.broadcast();

  AgentSessionImpl({
    required LLMProvider llmProvider,
    required List<AgentTool> tools,
    required String systemPrompt,
  }) : _llmProvider = llmProvider,
       _tools = tools,
       _systemPrompt = systemPrompt;

  @override
  ValueListenable<SessionState> get state => _state;

  @override
  Stream<AgentEvent> get events => _events.stream;

  @override
  Future<void> transform(String prompt, {List<BinaryContent>? attachments}) async {
    try {
      // 1. Update state to thinking and emit event
      _state.value = _state.value.copyWith(status: ActivityStatus.thinking);
      _events.add(ThinkingEvent('Processing your request...'));

      // 2. Build user message
      final userMessage = AgentMessage(
        id: Uuid().v4(),
        role: MessageRole.user,
        content: [
          TextContent(prompt),
          if (attachments != null) ...attachments,
        ],
        timestamp: DateTime.now(),
      );

      // 3. Append to history
      final newHistory = [..._state.value.history, userMessage];
      _state.value = _state.value.copyWith(history: newHistory);

      // 4. Call LLM
      final response = await _llmProvider.generateResponse(
        systemPrompt: _systemPrompt,
        history: newHistory,
        tools: _tools,
      );

      // 5. Process response
      await _processResponse(response.message);
    } catch (e, stackTrace) {
      // Handle errors gracefully
      _events.add(ErrorEvent('LLM request failed: $e', e));
      _state.value = _state.value.copyWith(
        status: ActivityStatus.idle,
        activeTool: null,
      );
      // Re-throw for caller awareness (optional - can be removed if events are sufficient)
      rethrow;
    }
  }

  Future<void> _processResponse(AgentMessage message) async {
    // Check for tool use
    for (final content in message.content) {
      if (content is ToolUseContent) {
        await _executeTool(content);
        return;
      }
    }

    // No tool call - update history and set idle
    final newHistory = [..._state.value.history, message];
    _state.value = _state.value.copyWith(
      history: newHistory,
      status: ActivityStatus.idle,
    );
  }

  Future<void> _executeTool(ToolUseContent toolUse) async {
    // Update state
    _state.value = _state.value.copyWith(
      status: ActivityStatus.calling_tool,
      activeTool: ToolCall(
        id: toolUse.id,
        name: toolUse.toolName,
        input: toolUse.input,
      ),
    );
    _events.add(ToolStartEvent(toolUse.toolName));

    try {
      // Find and execute tool
      final tool = _tools.firstWhereOrNull((t) => t.name == toolUse.toolName);
      if (tool == null) {
        throw ToolNotFoundException('Unknown tool: ${toolUse.toolName}');
      }

      final result = await tool.execute(toolUse.input);

      _events.add(ToolEndEvent(toolUse.toolName, !result.isError));

      // CRITICAL: Capture ChartConfiguration from tool result
      if (result.data is ChartConfiguration) {
        final chart = result.data as ChartConfiguration;
        _state.value = _state.value.copyWith(activeChart: chart);

        // Emit appropriate event based on tool name (see Section 7.5)
        if (toolUse.toolName == 'create_chart') {
          _events.add(ChartCreatedEvent(chart));
        } else if (toolUse.toolName == 'modify_chart') {
          _events.add(ChartUpdatedEvent(chart));
        }
      }

      // Build tool result message
      final toolResultMessage = AgentMessage(
        id: Uuid().v4(),
        role: MessageRole.tool,
        content: [
          ToolResultContent(
            toolUseId: toolUse.id,
            output: result.output,
            isError: result.isError,
          ),
        ],
        timestamp: DateTime.now(),
      );

      // Send result back to LLM for final response
      final newHistory = [..._state.value.history, toolResultMessage];
      _state.value = _state.value.copyWith(history: newHistory, activeTool: null);

      final finalResponse = await _llmProvider.generateResponse(
        systemPrompt: _systemPrompt,
        history: newHistory,
        tools: _tools,
      );

      await _processResponse(finalResponse.message);
    } catch (e, stackTrace) {
      // Tool execution failed - emit error event
      _events.add(ToolEndEvent(toolUse.toolName, false));
      _events.add(ErrorEvent('Tool execution failed: $e', e));
      _state.value = _state.value.copyWith(
        status: ActivityStatus.idle,
        activeTool: null,
      );
      rethrow;
    }
  }

/// Exception thrown when a requested tool is not found
class ToolNotFoundException implements Exception {
  final String message;
  ToolNotFoundException(this.message);
  @override
  String toString() => message;
}

  // --- Chart Update (User Edit Flow) ---
  // See decisions/chart-propagation.md Section "The Update Loop"

  @override
  void updateChart(ChartConfiguration newConfig) {
    // 1. Update state with new chart
    _state.value = _state.value.copyWith(activeChart: newConfig);

    // 2. Emit event for persistence layer
    _events.add(ChartUpdatedEvent(newConfig));

    // Note: The NEXT transform() call will automatically include
    // this chart as context because _buildContextualPrompt() reads
    // from state.activeChart.
  }

  // --- Cancellation Support ---

  CancelableOperation<LLMResponse>? _currentOperation;

  @override
  Future<void> cancel() async {
    // Cancel any in-progress LLM operation
    await _currentOperation?.cancel();
    _currentOperation = null;

    // Reset state to idle
    _state.value = _state.value.copyWith(
      status: ActivityStatus.idle,
      activeTool: null,
    );

    _events.add(const CancelledEvent());
  }

  // --- Resource Cleanup ---

  @override
  void dispose() {
    // Cancel any pending operation
    _currentOperation?.cancel();
    _currentOperation = null;

    // Close event stream
    _events.close();

    // Dispose value notifier
    _state.dispose();
  }
}

// Event for cancellation
class CancelledEvent extends AgentEvent {
  const CancelledEvent();
}
```

> **Implementation Note:** The `transform()` method should wrap the LLM call in a `CancelableOperation` and assign it to `_currentOperation` to enable cancellation. The `async` package provides `CancelableOperation`.

## 12. Implementation Rules

1. **NO DOMAIN LOGIC:** `braven_agent` does NOT know about "Athletes", "FIT Files", or "Projects". Those are App-layer concerns.

2. **IMMUTABLE STATE:** Always emit new `SessionState` objects. Never mutate.

3. **ERROR RECOVERY:** If LLM returns invalid JSON for a chart, catch the error in `CreateChartTool`, return `ToolResult(isError: true, output: "...")`. The LLM sees this and self-corrects.

4. **CHART CAPTURE:** The `AgentSession` MUST check `ToolResult.data` for `ChartConfiguration` objects and update `state.activeChart` + emit `ChartCreatedEvent`.

5. **DEFAULT COLORS:** If no color specified for a series, assign from the default palette.

6. **UUID GENERATION:** All charts get a unique `id` via `Uuid().v4()`.

7. **RENDERER IN BRAVEN_AGENT:** `ChartRenderer` lives in `braven_agent` because it owns the `ChartConfiguration` model. It outputs `BravenChartPlus` widgets:

   ```dart
   // Usage with braven_agent
   import 'package:braven_agent/braven_agent.dart';
   final widget = const ChartRenderer().render(myConfig);
   ```

8. **MODELS IN BRAVEN_AGENT:** `ChartConfiguration`, `SeriesConfig`, `XAxisConfig`, `YAxisConfig`, `AnnotationConfig` all live in `braven_agent`. They are specified in this document and implemented fresh.
