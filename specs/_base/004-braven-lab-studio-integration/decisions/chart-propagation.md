# Decisions: Chart State Propagation

## 6. Chart State Propagation

Charts are special. They are both **Content** (generated by Agent) and **Context** (edited by User).

### Where does the Chart live?

The chart MUST be part of the `SessionState` to ensure synchronization with the chat flow.

```dart
class SessionState {
  // ... other fields

  /// The chart currently relevant to the conversation.
  /// 1. If Agent just created one, it's here.
  /// 2. If User selects one from history, it's here.
  /// 3. If User edits it, the updated version is here.
  final ChartConfig? activeChart;
}
```

### The Update Loop

1.  **Agent Creates:** Agent calls `create_chart` -> `SessionState` updates with new `activeChart`. UI renders it.
2.  **User Edits:** User changes color in UI -> Calls `session.updateChart(newConfig)`.
3.  **State Update:** `SessionState` updates with the modified `activeChart`.
4.  **Agent Awareness:** The NEXT time the user sends a message, `BravenAgent` automatically grabs `session.state.activeChart` and injects it as context ("User is looking at chart X with these properties...").

**Decision:** Do NOT create a separate `Stream<Chart>`. Keep it locked inside `SessionState` to guarantee that the Chart the user sees on screen is _exactly_ the same Chart the Agent will see in the next turn.

## 7. Notification Strategy: UI vs Persistence

The answer to "How is the consumer notified?" has two parts, serving two different needs.

### Path A: The "Render Loop" (Instant UI Feedback)

**Mechanism:** `ValueListenable<SessionState>`
**Purpose:** Syncing the screen with the agent.
**How it works:**

1. Agent generates JSON -> Updates `state.activeChart`.
2. `ValueNotifier` fires.
3. The app's `ValueListenableBuilder` rebuilds.
4. The `BravenChart` widget redraws immediately with the new data.

### Path B: The "Side Effect Loop" (Persistence & Navigation)

**Mechanism:** `Stream<AgentEvent>`
**Purpose:** Saving data, showing toasts, changing tabs.
**How it works:**

1. Agent generates JSON -> Emits `ChartCreatedEvent(config)`.
2. App listener intercepts event.
3. **App Logic:**
   - "Oh, a new chart! Save this `ChartConfig` to the `ProjectDatabase` (SQLite)."
   - "Show a snackbar: 'Chart Ready'."
   - "If on mobile, switch from Chat Tab to Chart Tab."

### Summary

- **Use `state.activeChart`** to DRAW the chart.
- **Use `events`** to SAVE the chart or MOVE the screen.

## 8. Event Vocabulary

To distinguish between a completely new chart and an update to an existing one, the Agent uses specific events:

- `ChartCreatedEvent(config)`: The Agent has generated a _new_ visualization. The App should create a new database record.
- `ChartUpdatedEvent(config)`: The Agent has modified the _existing_ active chart (e.g., changed color, added series). The App should update the existing record.
  - _Note: `state.activeChart` handles the visual update automatically. This event is strictly for DB persistence triggers._
